Formalize Listing Requirements

It has become evident that we need to make our listing requirements more clear and robust in order to prevent the possibility of malicious tokens entering the system. We also need a formal document on GitHub that outlines the requirements for a proper token listing going forward.

Benefits:

This lowers the likelihood of malicious tokens entering the system. This increases the openness of our procedure and opens it up for community feedback. This formalizes a procedure, so people understand the requirements of a proper listing request. This makes it easier on token developers to give us the proper information required for a submission.

Why the required data:

A website will give much needed information about the token, team, and provide a source for verifying information given in the issue submission. Chrome will require all websites be HTTPS by June 2018. A description helps inform users about the token. We need a link to verify that the submitted contract address is the correct address for the project and not an imposter address. Public forums for discussion on the project allow the community to weigh in on the token's credence. This strengthens the case for the project's liveliness. Third party verification is another step that makes it easier to validate/verify the token issuer is serious about the project. We need to make sure the token issuer is reachable.

For Token Submissions

Anybody can request a token to be listed on ForkDelta, not just the token developers. However, the token submissions must strictly adhere to the following rules.

Absolutely required:

The contract address. A link to the token issuer's official website. This MUST be in HTTPS. 1-3 sentences for the token's description. A link to an official page confirming the above contract address. This needs to be on the official domain of the token issuer. Etherscan does not count as a confirmation. Links third-party review or discussion of at least two kinds. This can include, but is not limited to: comments from high ranked users in Bitcointalk, third party blogs, ICO ranking websites, established company partnerships, etc. Well-defined channels of communication with the token issuer must be published on the website. This includes, but is not limited to: public social media, some form of instant messaging (Telegram, Discord), email. You must pay attention to the issue or pull request you post. We will post comments and make updates on it, some of which will need your attention.

Optional questions:

Do you represent the token issuer or are you simply an interested member of the community? How was this token distributed? (ICO, Airdrop, etc.) Is there more information on the token issuer that is separate from the token? Example: Information about a parent company.

Strongly encouraged:

Presence of a whitepaper or a project roadmap. A link to a third-party review of the token like an ICO ranking website or a blog. CoinMarketCap listing Strong social media presence More than 30 token holders Team member information on website

Strongly discouraged:

Piggybacking off of other coins in your token names. For example: "BitcoinEther" Using free domains (like WIX, Weebly, etc)

Absolutely not allowed:

Same ticker as already listed project on ForkDelta Same ticker or name as prominent project Any form of plagiarization including, but not limited to: pictures, names, descriptions Affiliation with any other verified scams Falsified links Exceptions There may be very rare cases where prominent ICOs do not adhere to our requirements, but have a large amount of users and are legitimate projects. We reserve the right to waive requirements in those rare cases.
